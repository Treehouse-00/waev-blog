You plug in a second node, press send, and the message arrives. Simple enough. But how did it get there — and what happens differently the second time you send a message to the same person?

Most LoRa mesh newcomers assume the answer is "flooding." Every node hears it, every node rebroadcasts it, message reaches destination. That is how some systems work. MeshCore does something more considered.

<div class="callout"><p><strong>In short.</strong> How LoRa mesh routing works in MeshCore: the first message to an unknown destination floods through repeaters to discover a path, the destination sends that path back, and every subsequent message travels direct along the cached route. Only the repeaters on the learned path handle direct messages — all others stay silent. Group channel messages are the exception: they always flood.</p></div>

## The problem flooding solves — and creates

LoRa mesh networks have a routing challenge that wired networks do not: no node has a pre-configured map of who can hear whom. Topology shifts as nodes join, move, or lose power. The battery on a hilltop repeater drains overnight. Someone adds a new relay in a neighborhood that had a coverage gap.

Flooding is the honest answer to this uncertainty. If you do not know the route, send the message everywhere, and let it find its own way. It works. But it has a cost.

Every repeater that hears a flood packet retransmits it. A network with six repeaters and one incoming message can produce up to six retransmissions, each consuming airtime on the shared radio channel. Do that for every message, to every destination, on every send, and the channel fills up fast. At small scale this is fine. As the network grows, the math gets ugly.

The better approach: flood once to learn the route, then use the knowledge.

## Three phases

MeshCore routing for unicast messages — direct messages between two contacts — follows a three-phase lifecycle.

**Phase 1 — Discovery flood.** The sender has no cached route to the destination. It transmits the message with a flood route type. Every repeater that receives the packet checks whether it has already seen it (duplicate suppression prevents forwarding loops), then appends its own one-byte identifier to the path field and retransmits. The packet fans out across the network. When the destination receives it, the path field holds the complete record of every repeater the message passed through.

**Phase 2 — Path return.** The destination reverses the accumulated path and sends it back in a PATH packet. When the sender receives this, it stores the route — the ordered list of repeater identifiers that connect them.

**Phase 3 — Cached direct routing.** Every subsequent message to that contact uses the stored path. Each repeater on the path checks whether its identifier matches the first entry in the path field, strips its entry, and forwards to the next hop. Repeaters whose identifier is not in the path ignore the packet entirely. The airtime savings are significant: direct routing uses roughly 95–98% less airtime than flooding every message (source: EastMesh Wiki, MeshCore routing algorithm documentation).

## What "clients do not repeat" means in practice

This routing model only works cleanly because of a fundamental design constraint: companion radios — the firmware you run on the node connected to your phone — do not retransmit packets for others. Only repeaters relay traffic.

A common point of confusion for people coming from systems where every node is a relay: your handheld radio does not forward the messages it overhears. If you want a node to extend coverage, it needs to be running repeater firmware, not companion firmware. This is not a limitation — it is what keeps the network from becoming a wall of reflections.

One practical implication: the number of relaying nodes on your mesh is determined by how many dedicated repeaters you deploy, not by how many radios are on the network. Add fifty new users with companion radios and you add zero relay traffic. Add one well-placed repeater and you extend coverage.

## What this means for hop count

Because direct routing follows the cached path exactly, the hop count you see in Live Packets for a direct message reflects the actual learned route — not a flood path that happened to be short. A node that shows three hops reached your observer via three specific relays, in that order.

Hop-count creep — a node that showed two hops last week now showing four — means something on the shorter path degraded or disappeared, and the flood rediscovery found a longer alternative. Understanding the routing model makes this signal easier to interpret: it is not noise; it is the routing adapting to a topology change.

The SNR and hop count interpretation in [Reading the Signal](/blog/reading-the-signal/) applies directly here — those numbers are the observable output of this routing mechanism.

## Group channels are different

The flood-then-direct model applies to direct messages between two known contacts. Group channel messages — the public channel and any private group channels — always flood every time, because there is no single recipient to build a path toward. The channel key encrypts the payload; only nodes with that key can read it; but the routing itself is a broadcast flood.

This distinction matters for airtime management. A busy group channel on a network with many relays consumes proportionally more airtime than direct message traffic. For most community deployments the public channel is a coordination layer, not a high-volume data pipe, so this is rarely a bottleneck — but it is worth knowing where the accounting lands.
